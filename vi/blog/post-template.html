Tối ưu hóa tốc độ suy luận Transformer bằng CUDA Kernel tùy chỉnh

2026-03-15 · Minh Tran, Alex Rivera · 12 phút đọc

Khi tụi mình bắt tay vào tối ưu hóa quá trình suy luận (inference) cho model FTU-7B, các bài đo hiệu năng đầu tiên cho thấy phần Attention mặc định của PyTorch chính là điểm nghẽn lớn nhất — nó chiếm khoảng 60% tổng thời gian suy luận trên GPU A100. Bài viết này sẽ mô tả cách tụi mình tự viết các CUDA kernel tùy chỉnh để tăng tốc độ lên gấp 3 lần, cùng với đó là những bài học xương máu rút ra được trong quá trình thực hiện.

Vấn đề: Attention bị giới hạn bởi bộ nhớ (Memory-Bound)

Cơ chế Attention trong kiến trúc Transformer về bản chất là một tác vụ bị giới hạn bởi bộ nhớ (memory-bound). Cách triển khai thông thường sẽ phải tạo ra toàn bộ ma trận Attention kích thước N×N, yêu cầu bộ nhớ và băng thông ở mức O(N²). Với các chuỗi dài (trên 4000 token), đây trở thành chi phí chiếm tỷ trọng lớn nhất — không phải do năng lực tính toán mà do cách dữ liệu di chuyển giữa các phân cấp bộ nhớ (Memory Hierarchy) của GPU.

Quá trình thu thập dữ liệu (Profiling) cho thấy Attention mặc định dành phần lớn thời gian để chờ đợi dữ liệu được truyền đi thay vì thực hiện các phép tính có ích. Mật độ tính toán thực tế thấp hơn rất nhiều so với những gì card A100 có thể đáp ứng: 

# Kết quả đo hiệu năng (A100 80GB, batch=1, seq_len=4096)
Standard Attention:     142ms  (băng thông bộ nhớ: 1.2 TB/s)
cuBLAS GEMM (Q×K):      38ms
Softmax:                 24ms
cuBLAS GEMM (A×V):      36ms
Cấp phát bộ nhớ:         44ms  ← chi phí ẩn

Con số 44ms chỉ dành riêng cho việc cấp phát bộ nhớ là một dấu hiệu cảnh báo đỏ. Mỗi lần chạy qua (forward pass) đều phải cấp phát và giải phóng các vùng nhớ tạm cho ma trận Attention — đây là một mô hình đã được tối ưu hóa trong Flash Attention nhưng vẫn tồn tại trong nhiều cách triển khai thông thường.

Giải pháp: Kiến trúc Fused Tiled Attention

Thay vì tính toán Attention qua từng giai đoạn riêng biệt (như nhân ma trận GEMM -> tính Softmax -> nhân ma trận GEMM), tụi mình gộp tất cả vào một kernel duy nhất xử lý theo từng khối (tile) dữ liệu đầu vào. Cách làm này tương tự như Flash Attention, nhưng có một vài chỉnh sửa riêng biệt cho kiến trúc mô hình của tụi mình:

  • Attention theo nhóm truy vấn (Grouped-query attention - GQA): Model FTU-7B dùng 8 đầu KV (KV heads) chia sẻ cho 32 đầu truy vấn (query heads), yêu cầu việc quản lý bộ nhớ chia sẻ (shared memory) phải cực kỳ cẩn thận.
  • Tích hợp mã hóa vị trí xoay (Rotary Position Embedding - RoPE): Tụi mình tính toán vị trí ngay trong kernel thay vì để nó là một bước tiền xử lý riêng biệt.
  • Tính Softmax trực tiếp (Online Softmax) với bộ cộng dồn FP32: Duy trì sự ổn định về mặt số học trong khi vẫn giữ luồng dữ liệu ở định dạng FP16 hoặc BF16.

Chi tiết triển khai

Ý tưởng cốt lõi là chia nhỏ phép tính theo khối sao cho mỗi khối luồng (thread block) xử lý một nhóm vị trí truy vấn đối với toàn bộ các cặp Key-Value, sau đó cộng dồn kết quả đầu ra vào bộ nhớ chia sẻ: 

// Cấu trúc kernel rút gọn (bản thực tế dài khoảng 800 dòng)
__global__ void fused_attention_kernel(
    const half* __restrict__ Q,
    const half* __restrict__ K,
    const half* __restrict__ V,
    half* __restrict__ O,
    const int seq_len,
    const int head_dim
) {
    // Mỗi khối xử lý BLOCK_M vị trí truy vấn
    const int q_start = blockIdx.x * BLOCK_M;

    // Bộ nhớ chia sẻ cho các khối Q, K, V
    __shared__ half smem_q[BLOCK_M][HEAD_DIM];
    __shared__ half smem_k[BLOCK_N][HEAD_DIM];
    __shared__ half smem_v[BLOCK_N][HEAD_DIM];

    // Trạng thái cho Online Softmax
    float row_max = -INFINITY;
    float row_sum = 0.0f;
    float acc[BLOCK_M][HEAD_DIM] = {0};

    // Lặp qua các khối Key và Value
    for (int kv_start = 0; kv_start < seq_len; kv_start += BLOCK_N) {
        load_tile(K, smem_k, kv_start);
        load_tile(V, smem_v, kv_start);
        __syncthreads();

        // Tính QK^T, áp dụng RoPE, tỷ lệ hóa, tạo mask
        // Cập nhật Online Softmax
        // Cộng dồn O = softmax(QK^T) * V
    }

    store_output(O, acc, row_sum, q_start);
}

Tụi mình biên dịch sẵn kernel để tránh việc hệ thống phải biên dịch lại (JIT) khi đưa vào vận hành thực tế: 

# Biên dịch cho A100 (SM 8.0) và H100 (SM 9.0)
nvcc -O3 -arch=sm_80 -gencode=arch=compute_90,code=sm_90 \
     --use_fast_math -lineinfo \
     -o fused_attn.so --shared fused_attention.cu

# Chạy quy trình đo hiệu năng
python bench.py --kernel fused --seq-lengths 2048 4096 8192 16384

Kết quả

Sau nhiều lần tinh chỉnh kích thước khối dữ liệu, cách sắp xếp bộ nhớ chia sẻ và lập lịch hoạt động, tụi mình đã đạt được những cải thiện đáng kể trên mọi độ dài chuỗi:

Bài học quan trọng nhất không nằm ở thuật toán — mà là ở việc hiểu sâu về kiến trúc bộ nhớ của GPU để làm sao các đơn vị tính toán (SM) luôn được "nuôi" đầy đủ dữ liệu trong mọi chu kỳ.

Trên hệ thống đo chuẩn (A100 80GB, định dạng BF16, kích thước mẫu bằng 1):

  • Chuỗi 2048: Nhanh gấp 2.1 lần (68ms còn 32ms)
  • Chuỗi 4096: Nhanh gấp 2.8 lần (142ms còn 51ms)
  • Chuỗi 8192: Nhanh gấp 3.2 lần (412ms còn 129ms)
  • Chuỗi 16384: Nhanh gấp 3.4 lần (1.6 giây còn 470ms)

Đóng gói triển khai

Để môi trường suy luận có thể chạy ổn định trên các máy khác nhau, tụi mình đóng gói mọi thứ bằng Docker với quy trình build nhiều giai đoạn (multi-stage build): 

# Giai đoạn 1: Biên dịch CUDA kernel
FROM nvidia/cuda:12.4.1-devel-ubuntu22.04 AS builder
WORKDIR /build
COPY kernels/ ./kernels/
RUN nvcc -O3 -arch=sm_80 --use_fast_math \
        -o fused_attn.so --shared kernels/fused_attention.cu

# Giai đoạn 2: Image runtime gọn nhẹ
FROM nvidia/cuda:12.4.1-runtime-ubuntu22.04
RUN apt-get update && apt-get install -y --no-install-recommends \
        python3 python3-pip && \
    rm -rf /var/lib/apt/lists/*

WORKDIR /app
COPY --from=builder /build/fused_attn.so /app/lib/
COPY requirements.txt .
RUN pip3 install --no-cache-dir -r requirements.txt
COPY src/ ./src/

EXPOSE 8000
CMD ["python3", "-m", "uvicorn", "src.api:app", "--host", "0.0.0.0", "--port", "8000"]
# Build và chạy
docker build -t ftu-inference .
docker run --gpus all -p 8000:8000 ftu-inference

Bài học kinh nghiệm

Viết CUDA kernel vừa là một môn khoa học, vừa là một môn nghệ thuật. Đây là những bài học "ngầm" mà tụi mình rút ra được:

  1. Hãy đo hiệu năng trước khi tối ưu hóa: Ban đầu tụi mình cứ tưởng các phép nhân ma trận (GEMM) là điểm nghẽn. Sự thật không phải vậy — vấn đề nằm ở việc cấp phát bộ nhớ và cách sắp xếp dữ liệu.
  2. Xung đột trong bộ nhớ chia sẻ (Bank Conflict) là có thật: Một thay đổi nhỏ chỉ 2 dòng code để căn chỉnh bộ nhớ (thêm phần đệm để tránh xung đột) đã giúp tăng tốc thêm 15%.
  3. Kiểm tra sự ổn định về mặt số học một cách gắt gao: Việc cộng dồn với độ chính xác FP16 đã tạo ra những sai số nhỏ, chỉ xuất hiện với các chuỗi rất dài. Vì vậy, việc dùng bộ cộng dồn FP32 là điều bắt buộc.

Toàn bộ mã nguồn triển khai kernel này hiện có sẵn tại FlashKernel repository. Tụi mình rất mong nhận được đóng góp và các bài test hiệu năng trên các dòng GPU khác từ cộng đồng.
