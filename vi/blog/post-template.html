Tăng tốc Transformer Inference với Custom CUDA Kernels

2026-03-15 · Minh Tran, Alex Rivera · 12 phút đọc

Khi chúng tôi bắt đầu tối ưu hóa inference cho FTU-7B, các benchmark ban đầu cho thấy attention tiêu chuẩn trong PyTorch là nút thắt cổ chai chính — chiếm khoảng 60% tổng thời gian inference trên GPU A100. Bài viết này mô tả cách chúng tôi viết custom CUDA kernel để đạt tốc độ nhanh gấp 3 lần, và những bài học rút ra trong quá trình đó.

Vấn đề: Attention bị giới hạn bởi bộ nhớ

Transformer attention về cơ bản là một phép toán bị giới hạn bởi bộ nhớ. Cách triển khai tiêu chuẩn tạo ra toàn bộ ma trận attention N×N, yêu cầu bộ nhớ và băng thông O(N²). Với các chuỗi dài (4K+ token), đây trở thành chi phí chính — không phải vì tính toán, mà vì cách dữ liệu di chuyển giữa các tầng bộ nhớ của GPU.

Kết quả profiling cho thấy attention tiêu chuẩn dành phần lớn thời gian chờ truyền dữ liệu bộ nhớ thay vì thực hiện tính toán hữu ích. Cường độ tính toán (arithmetic intensity) thấp hơn nhiều so với khả năng của A100: 

# Profiling results (A100 80GB, batch=1, seq_len=4096)
Standard Attention:     142ms  (memory bandwidth: 1.2 TB/s)
cuBLAS GEMM (Q×K):      38ms
Softmax:                 24ms
cuBLAS GEMM (A×V):      36ms
Memory allocation:       44ms  ← the hidden cost

44ms dành riêng cho cấp phát bộ nhớ là một dấu hiệu đáng lo ngại. Mỗi forward pass đều cấp phát và giải phóng các buffer tạm cho ma trận attention — một pattern đã được tối ưu trong Flash Attention nhưng vẫn tồn tại trong nhiều cách triển khai tiêu chuẩn.

Cách tiếp cận: Fused Tiled Attention

Thay vì tính attention qua các bước riêng biệt GEMM → softmax → GEMM, chúng tôi gộp tất cả vào một kernel duy nhất hoạt động trên các tile của đầu vào. Về mặt khái niệm, cách này tương tự Flash Attention, nhưng với một số điều chỉnh riêng cho kiến trúc mô hình của chúng tôi:

  • Grouped-query attention (GQA) — FTU-7B sử dụng 8 KV head chia sẻ cho 32 query head, đòi hỏi quản lý shared memory cẩn thận
  • Tích hợp RoPE — Chúng tôi tính rotary position embedding ngay trong kernel thay vì thực hiện ở bước tiền xử lý riêng
  • Online softmax với FP32 accumulator — Duy trì độ ổn định số học trong khi giữ đường dẫn dữ liệu ở FP16/BF16

Chi tiết triển khai

Ý tưởng cốt lõi là chia nhỏ phép tính thành các tile để mỗi thread block xử lý một khối query với tất cả các cặp key-value, tích lũy kết quả đầu ra trong shared memory: 

// Simplified kernel structure (actual implementation is ~800 lines)
__global__ void fused_attention_kernel(
    const half* __restrict__ Q,
    const half* __restrict__ K,
    const half* __restrict__ V,
    half* __restrict__ O,
    const int seq_len,
    const int head_dim
) {
    // Each block handles BLOCK_M query positions
    const int q_start = blockIdx.x * BLOCK_M;

    // Shared memory for Q, K, V tiles
    __shared__ half smem_q[BLOCK_M][HEAD_DIM];
    __shared__ half smem_k[BLOCK_N][HEAD_DIM];
    __shared__ half smem_v[BLOCK_N][HEAD_DIM];

    // Online softmax state
    float row_max = -INFINITY;
    float row_sum = 0.0f;
    float acc[BLOCK_M][HEAD_DIM] = {0};

    // Iterate over K,V blocks
    for (int kv_start = 0; kv_start < seq_len; kv_start += BLOCK_N) {

Bài học rút ra

Viết CUDA kernel vừa là nghệ thuật vừa là khoa học. Dưới đây là những bài học không hiển nhiên từ dự án này:

  1. Profile trước khi tối ưu. Giả định ban đầu của chúng tôi là các phép toán GEMM là nút thắt cổ chai. Thực tế không phải — cấp phát bộ nhớ và bố cục dữ liệu mới là vấn đề.
  2. Shared memory bank conflict là có thật. Chỉ thay đổi 2 dòng trong bố cục bộ nhớ (thêm padding để tránh bank conflict) đã cho chúng tôi tăng tốc 15%.
  3. Kiểm tra độ chính xác số học một cách kỹ lưỡng. Phép tích lũy FP16 gây ra các lỗi tinh vi chỉ xuất hiện ở các chuỗi dài. FP32 accumulator là bắt buộc, không thỏa hiệp.

Toàn bộ mã nguồn kernel có sẵn trong repository FlashKernel của chúng tôi. Chúng tôi hoan nghênh đóng góp và benchmark trên các kiến trúc GPU khác.
