Cẩm nang sinh tồn cho sinh viên CNTT tại FTU

2026-03-28 · FTU Labs · 20 phút đọc

Bài viết này bao gồm những kỹ năng thiết yếu mà mọi sinh viên CNTT tại FTU nên học — từ cài đặt IDE đầu tiên đến triển khai một dự án hoàn chỉnh. Mỗi phần ngắn gọn, thực tế và đầy đủ tài liệu tham khảo. Hãy lưu trang này lại và quay lại khi bạn cần.

1. Cài đặt IDE

VS Code là trình soạn thảo được khuyên dùng. Miễn phí, nhiều extension, và chạy trên mọi hệ điều hành. 

# Install VS Code (Ubuntu/Debian)
sudo apt update && sudo apt install code

# macOS (via Homebrew)
brew install --cask visual-studio-code

Các extension cần thiết — cài từ thanh Extensions (Ctrl+Shift+X):

  • Python — IntelliSense, linting, debugging
  • Pylance — kiểm tra kiểu dữ liệu nhanh
  • GitLens — xem git blame và lịch sử ngay trong code
  • Prettier — tự động format HTML/CSS/JS
  • Remote - SSH — chỉnh sửa file trên server từ xa
  • Live Server — hot-reload cho dự án HTML/CSS

Phím tắt hữu ích:

Ctrl+P           # Quick file open
Ctrl+Shift+P     # Command palette
Ctrl+`            # Toggle terminal
Ctrl+D            # Select next occurrence
Alt+Up/Down       # Move line up/down
Ctrl+Shift+K      # Delete line

2. Môi trường ảo

Đừng bao giờ cài package toàn cục. Hãy dùng virtual environment để cô lập dependency cho từng dự án.

Lựa chọn A: venv (có sẵn)

# Create
python -m venv .venv

# Activate
source .venv/bin/activate    # Linux/macOS
.venv\Scripts\activate       # Windows

# Install packages
pip install numpy pandas torch

# Freeze dependencies
pip freeze > requirements.txt

# Reproduce elsewhere
pip install -r requirements.txt

# Deactivate
deactivate

Lựa chọn B: conda (cho data science / CUDA)

# Install Miniconda (lightweight)
# https://docs.conda.io/en/latest/miniconda.html

# Create environment
conda create -n myproject python=3.11 -y
conda activate myproject

# Install with conda-forge
conda install -c conda-forge numpy pandas

# Export
conda env export > environment.yml

# Reproduce
conda env create -f environment.yml

Lựa chọn C: uv (nhanh, hiện đại)

# Install uv
curl -LsSf https://astral.sh/uv/install.sh | sh

# Create project with virtual env
uv init myproject && cd myproject
uv venv

# Add dependencies (auto-resolves)
uv add numpy pandas torch

# Sync (install everything from pyproject.toml)
uv sync
Nguyên tắc chung: dùng venv cho dự án đơn giản, conda khi cần CUDA/thư viện hệ thống, và uv khi bạn muốn tốc độ.

Tài liệu tham khảo:

3. GitHub, Code sạch & Đóng góp

3.1 Tạo GitHub Personal Access Token

# Go to: GitHub → Settings → Developer settings → Personal access tokens → Fine-grained tokens
# Select permissions: Contents (read/write), Pull requests (read/write)
# Copy the token

# Configure git (one-time)
git config --global user.name "Your Name"
git config --global user.email "you@example.com"

# Clone with token (HTTPS)
git clone https://<TOKEN>@github.com/ftulabs/your-repo.git

# Or use SSH (recommended) — generate a key:
ssh-keygen -t ed25519 -C "you@example.com"
cat ~/.ssh/id_ed25519.pub
# Add the public key at: GitHub → Settings → SSH keys

3.2 Quy trình Git cơ bản

# Start from latest main
git checkout main && git pull

# Create feature branch
git checkout -b feature/add-login-page

# Work, stage, commit
git add src/login.py tests/test_login.py
git commit -m "Add login page with form validation"

# Push and create PR
git push -u origin feature/add-login-page
gh pr create --title "Add login page" --body "Implements #12"

3.3 Viết code sạch

# BAD — no context, magic numbers, no types
def calc(x):
    return x * 1.1

# GOOD — descriptive name, type hints, docstring
def calculate_price_with_tax(base_price: float, tax_rate: float = 0.1) -> float:
    """Return the total price including tax."""
    return base_price * (1 + tax_rate)

Nguyên tắc viết comment:

  • Comment tại sao, không phải cái gì. Code cho biết nó làm gì; comment giải thích mục đích.
  • Dùng docstring cho hàm public. Hạn chế comment inline.
  • Xoá code bị comment — đó là việc của git history.

3.4 Quy trình đóng góp (cho các repo FTU Labs)

  1. Fork repo (hoặc clone nếu bạn có quyền ghi)
  2. Tạo branch từ main với tên mô tả rõ ràng
  3. Viết code + test — theo style hiện có
  4. Commit với message rõ ràng (tiền tố feat:, fix:, docs:)
  5. Push và mở Pull Request
  6. Xử lý feedback từ review, sau đó merge 
# Conventional commit examples
git commit -m "feat: add user authentication endpoint"
git commit -m "fix: resolve race condition in queue worker"
git commit -m "docs: update API reference for v2 routes"
git commit -m "refactor: extract validation logic into utils"

4. LaTeX, Overleaf & arXiv

4.1 Cơ bản về LaTeX

LaTeX là chuẩn để viết bài báo khoa học. Dùng Overleaf để cộng tác chỉnh sửa (không cần cài đặt trên máy). 

\documentclass{article}
\usepackage[utf8]{inputenc}
\usepackage{amsmath, amssymb, graphicx, hyperref}

\title{Your Paper Title}
\author{Your Name \\ FTU Labs}
\date{\today}

\begin{document}
\maketitle
\begin{abstract}
  A concise summary of your work in 150–250 words.
\end{abstract}

\section{Introduction}
Your introduction text here. Cite with~\cite{vaswani2017attention}.

\section{Method}
An equation:
\begin{equation}
  \text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V
\end{equation}

A figure:
\begin{figure}[h]
  \centering
  \includegraphics[width=0.8\textwidth]{figures/architecture.png}
  \caption{Model architecture overview.}
  \label{fig:arch}
\end{figure}

\bibliographystyle{plain}
\bibliography{references}
\end{document}

4.2 Mẹo dùng Overleaf

  • Dùng Overleaf Git integration — clone dự án Overleaf qua git, chỉnh sửa trên máy, push lại
  • Bắt đầu từ template hội nghị: NeurIPS, ICML, ACL, AAAI đều có template trên Overleaf
  • Dùng \input{sections/method.tex} để chia bài báo dài thành nhiều file
  • Quản lý tài liệu tham khảo bằng file .bib — export từ Google Scholar hoặc Semantic Scholar

4.3 Nộp bài lên arXiv

# Prepare submission directory
mkdir arxiv_submission && cd arxiv_submission

# Copy source files (not the PDF)
cp main.tex references.bib figures/ -r .

# Flatten \input{} includes if needed
latexpand main.tex > merged.tex

# Create tarball
tar -czf submission.tar.gz *.tex *.bib *.bst figures/

# Upload at https://arxiv.org/submit
# Choose category (cs.CL, cs.LG, cs.CV, etc.)
# arXiv compiles the LaTeX on their servers

  • Docusaurus — phù hợp cho dự án nhiều tài liệu (dựa trên React)

# Example: deploy with GitHub Actions (add .github/workflows/deploy.yml)
name: Deploy
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - run: npm ci && npm run build
      - uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./build

6.3 MCP (Model Context Protocol) server

MCP cho phép trợ lý AI kết nối với các công cụ bên ngoài — database, API, hệ thống file, trình duyệt. Hãy xem nó như plugin cho AI. 

// .claude/settings.json — example MCP config
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@anthropic-ai/mcp-filesystem", "/path/to/allowed/dir"]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@anthropic-ai/mcp-github"],
      "env": { "GITHUB_TOKEN": "ghp_xxx" }
    }
  }
}

7. Học System Design

System design là kỹ năng thiết kế phần mềm có khả năng mở rộng. Kể cả khi bạn không đi phỏng vấn, hiểu những pattern này sẽ giúp bạn trở thành kỹ sư giỏi hơn.

7.1 Các khái niệm cốt lõi (theo thứ tự)

  1. Mô hình Client-Server — HTTP, REST API, chu trình request/response
  2. Database — SQL vs NoSQL, indexing, replication, sharding
  3. Caching — Redis, CDN, chiến lược cache invalidation
  4. Load balancing — round-robin, consistent hashing, health check
  5. Message queue — Kafka, RabbitMQ, pattern pub/sub
  6. Microservices vs monolith — đánh đổi, khi nào nên tách
  7. Định lý CAP — consistency, availability, partition tolerance

7.2 Cách luyện tập

# For each system design problem, follow this framework:

1. REQUIREMENTS     → What does the system do? (functional + non-functional)
2. ESTIMATION       → How much data? How many users? QPS?
3. HIGH-LEVEL DESIGN → Draw boxes and arrows (API, services, DB, cache)
4. DETAILED DESIGN  → Zoom into the hardest part (data model, algorithms)
5. BOTTLENECKS      → What breaks at scale? How to mitigate?

# Example: design a URL shortener
- Functional: shorten URL, redirect, analytics
- Scale: 100M URLs, 1000 writes/s, 10K reads/s
- Components: API gateway → shortening service → DB (key-value)
- Key decision: base62 encoding vs hash, read-heavy = cache layer

8. Dự án End-to-End: Từ ý tưởng đến CI/CD

Đây là nơi mọi thứ kết hợp lại. Sau đây là vòng đời đầy đủ của một dự án thực tế.

Giai đoạn 1: Ý tưởng & Lên kế hoạch

# Define the problem
- What problem are you solving?
- Who is the user?
- What exists already? (literature review / competitor analysis)

# Write a one-pager
- Problem statement (2–3 sentences)
- Proposed solution
- Success metrics
- Tech stack choices with justification

# Create a GitHub repo with structure
my-project/
├── README.md            # What, why, how to run
├── pyproject.toml       # Dependencies and metadata
├── src/
│   └── my_project/
│       ├── __init__.py
│       ├── model.py
│       └── api.py
├── tests/
│   └── test_model.py
├── docs/
├── .github/
│   └── workflows/
│       └── ci.yml
└── .gitignore

Giai đoạn 2: Thiết kế hệ thống

# Draw your architecture (use draw.io, Excalidraw, or Mermaid)

# Example: ML inference service
┌─────────┐     ┌───────────┐     ┌──────────┐
│  Client  │────▶│  FastAPI   │────▶│  Model   │
│ (React)  │◀────│  Server    │◀────│ (PyTorch)│
└─────────┘     └─────┬─────┘     └──────────┘
                      │
                ┌─────▼─────┐
                │ PostgreSQL │
                └───────────┘

# Define API contracts early
# POST /predict
{
  "input": "text to classify",
  "model_version": "v2"
}
# Response: { "label": "positive", "confidence": 0.94 }

Giai đoạn 3: Triển khai code

# Use branches for each feature
git checkout -b feat/model-inference

# Write tests FIRST (or alongside)
# tests/test_model.py
def test_predict_returns_label():
    result = predict("hello world")
    assert result["label"] in ["positive", "negative", "neutral"]
    assert 0 <= result["confidence"] <= 1

# Run tests locally before pushing
pytest tests/ -v

Giai đoạn 4: Pipeline CI/CD

# .github/workflows/ci.yml
name: CI
on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - run: pip install -e ".[dev]"
      - run: ruff check src/
      - run: pytest tests/ -v --tb=short

  deploy:
    needs: test
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to server
        run: |
          echo "Deploy step here"
          # e.g., docker build, push to registry, deploy to cloud
