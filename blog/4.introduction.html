Survival Guide for CS Students in FTU

2026-03-28 · FTU Labs · 20 min read

This guide covers the essential skills every CS student at FTU should learn — from setting up your first IDE to shipping an end-to-end project. Each section is short, practical, and packed with resources. Bookmark this page and revisit it as you progress.

1. Setting Up Your IDE

VS Code is the recommended editor. It's free, extensible, and works on every OS. 

# Install VS Code (Ubuntu/Debian)
sudo apt update && sudo apt install code

# macOS (via Homebrew)
brew install --cask visual-studio-code

Essential extensions — install from the Extensions sidebar (Ctrl+Shift+X):

  • Python — IntelliSense, linting, debugging
  • Pylance — fast type checking
  • GitLens — inline git blame and history
  • Prettier — auto-format HTML/CSS/JS
  • Remote - SSH — edit files on remote servers
  • Live Server — hot-reload for HTML/CSS projects

Useful shortcuts:

Ctrl+P           # Quick file open
Ctrl+Shift+P     # Command palette
Ctrl+`            # Toggle terminal
Ctrl+D            # Select next occurrence
Alt+Up/Down       # Move line up/down
Ctrl+Shift+K      # Delete line

2. Virtual Environments

Never install packages globally. Use virtual environments to isolate dependencies per project.

Option A: venv (built-in)

# Create
python -m venv .venv

# Activate
source .venv/bin/activate    # Linux/macOS
.venv\Scripts\activate       # Windows

# Install packages
pip install numpy pandas torch

# Freeze dependencies
pip freeze > requirements.txt

# Reproduce elsewhere
pip install -r requirements.txt

# Deactivate
deactivate

Option B: conda (for data science / CUDA)

# Install Miniconda (lightweight)
# https://docs.conda.io/en/latest/miniconda.html

# Create environment
conda create -n myproject python=3.11 -y
conda activate myproject

# Install with conda-forge
conda install -c conda-forge numpy pandas

# Export
conda env export > environment.yml

# Reproduce
conda env create -f environment.yml

Option C: uv (fast, modern)

# Install uv
curl -LsSf https://astral.sh/uv/install.sh | sh

# Create project with virtual env
uv init myproject && cd myproject
uv venv

# Add dependencies (auto-resolves)
uv add numpy pandas torch

# Sync (install everything from pyproject.toml)
uv sync
Rule of thumb: use venv for simple projects, conda when you need CUDA/system libs, and uv when you want speed.

  • Python venv docs

3. GitHub, Clean Code & Contributing

3.1 Create a GitHub Personal Access Token

# Go to: GitHub → Settings → Developer settings → Personal access tokens → Fine-grained tokens
# Select permissions: Contents (read/write), Pull requests (read/write)
# Copy the token

# Configure git (one-time)
git config --global user.name "Your Name"
git config --global user.email "you@example.com"

# Clone with token (HTTPS)
git clone https://<TOKEN>@github.com/ftulabs/your-repo.git

# Or use SSH (recommended) — generate a key:
ssh-keygen -t ed25519 -C "you@example.com"
cat ~/.ssh/id_ed25519.pub
# Add the public key at: GitHub → Settings → SSH keys

3.2 Git workflow basics

# Start from latest main
git checkout main && git pull

# Create feature branch
git checkout -b feature/add-login-page

# Work, stage, commit
git add src/login.py tests/test_login.py
git commit -m "Add login page with form validation"

# Push and create PR
git push -u origin feature/add-login-page
gh pr create --title "Add login page" --body "Implements #12"

3.3 Writing clean code

# BAD — no context, magic numbers, no types
def calc(x):
    return x * 1.1

# GOOD — descriptive name, type hints, docstring
def calculate_price_with_tax(base_price: float, tax_rate: float = 0.1) -> float:
    """Return the total price including tax."""
    return base_price * (1 + tax_rate)

Comment rules:

  • Comment why, not what. The code tells you what; comments explain intent.
  • Use docstrings for public functions. Use inline comments sparingly.
  • Delete commented-out code — that's what git history is for.

3.4 Contribution process (for FTU Labs repos)

  1. Fork the repo (or clone if you have write access)
  2. Create a branch from main with a descriptive name
  3. Write code + tests — follow existing style
  4. Commit with clear messages (feat:, fix:, docs: prefixes)
  5. Push and open a Pull Request
  6. Address review feedback, then merge 
# Conventional commit examples
git commit -m "feat: add user authentication endpoint"
git commit -m "fix: resolve race condition in queue worker"
git commit -m "docs: update API reference for v2 routes"
git commit -m "refactor: extract validation logic into utils"

4. LaTeX, Overleaf & arXiv

4.1 LaTeX basics

LaTeX is the standard for writing academic papers. Use Overleaf for collaborative editing (no local install needed). 

\documentclass{article}
\usepackage[utf8]{inputenc}
\usepackage{amsmath, amssymb, graphicx, hyperref}

\title{Your Paper Title}
\author{Your Name \\ FTU Labs}
\date{\today}

\begin{document}
\maketitle
\begin{abstract}
  A concise summary of your work in 150–250 words.
\end{abstract}

\section{Introduction}
Your introduction text here. Cite with~\cite{vaswani2017attention}.

\section{Method}
An equation:
\begin{equation}
  \text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V
\end{equation}

A figure:
\begin{figure}[h]
  \centering
  \includegraphics[width=0.8\textwidth]{figures/architecture.png}
  \caption{Model architecture overview.}
  \label{fig:arch}
\end{figure}

\bibliographystyle{plain}
\bibliography{references}
\end{document}

4.2 Tips for Overleaf

  • Use Overleaf Git integration — clone your Overleaf project via git, edit locally, push back
  • Start from conference templates: NeurIPS, ICML, ACL, AAAI all have Overleaf templates
  • Use \input{sections/method.tex} to split large papers into files
  • Manage references with .bib files — export from Google Scholar or Semantic Scholar

4.3 Submitting to arXiv

# Prepare submission directory
mkdir arxiv_submission && cd arxiv_submission

# Copy source files (not the PDF)
cp main.tex references.bib figures/ -r .

# Flatten \input{} includes if needed
latexpand main.tex > merged.tex

# Create tarball
tar -czf submission.tar.gz *.tex *.bib *.bst figures/

# Upload at https://arxiv.org/submit
# Choose category (cs.CL, cs.LG, cs.CV, etc.)
# arXiv compiles the LaTeX on their servers

5. Creating a Project Website

Every serious project deserves a landing page. GitHub Pages is the easiest way — free hosting, custom domain support, deploys on push.

5.1 Static HTML (simplest)

# Create repo: username.github.io or project-name
mkdir my-project-site && cd my-project-site
git init

# Create index.html
cat <<'HTML' > index.html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <title>My Project</title>
  <style>
    body { font-family: system-ui; max-width: 800px; margin: 2rem auto; padding: 0 1rem; }
  </style>
</head>
<body>
  <h1>Project Name</h1>
  <p>One-line description of what it does.</p>
  <h2>Demo</h2>
  <img src="demo.gif" alt="demo">
  <h2>Quick Start</h2>
  <pre>pip install my-project</pre>
</body>
</html>
HTML

# Push and enable GitHub Pages in repo Settings → Pages → main branch

5.2 Using a static site generator

  • Docusaurus — great for docs-heavy projects (React-based)
  • Jekyll — GitHub Pages native support, Markdown-based
  • Hugo — fastest build times, Go-based
  • Nerfies template — popular academic project page template 
# Example: deploy with GitHub Actions (add .github/workflows/deploy.yml)
name: Deploy
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - run: npm ci && npm run build
      - uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./build

6. AI Tools & Productivity Setup

6.1 Google Colab

Free GPU/TPU notebooks. Perfect for experiments before committing to local setup.

from google.colab import drive

Pro tip: use Colab Pro for longer runtimes and A100 access. Connect Colab to your local runtime via jupyter_http_over_ws for heavier workloads.

6.2 AI coding assistants

  • GitHub Copilot — free for students via GitHub Student Developer Pack. Install the VS Code extension.
  • Claude Code — Anthropic's CLI agent. Reads your codebase, writes code, runs commands. 
# Install Claude Code
npm install -g @anthropic-ai/claude-code

# Run in your project directory
cd my-project
claude

# Example usage inside Claude Code
> "Add unit tests for the auth module"
> "Refactor this function to use async/await"
> "Explain what this CUDA kernel does"

6.3 MCP (Model Context Protocol) servers

MCP lets AI assistants connect to external tools — databases, APIs, file systems, browsers. Think of it as plugins for AI. 

// .claude/settings.json — example MCP config
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@anthropic-ai/mcp-filesystem", "/path/to/allowed/dir"]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@anthropic-ai/mcp-github"],
      "env": { "GITHUB_TOKEN": "ghp_xxx" }
    }
  }
}

6.4 Other useful tools

  • Cursor — AI-native code editor (VS Code fork with built-in AI)
  • WandB — experiment tracking for ML (pip install wandb)
  • DVC — version control for data and models (pip install dvc)
  • pre-commit — auto-format and lint on every commit

7. Learning System Design

System design is the skill of architecting software that scales. Even if you're not interviewing, understanding these patterns makes you a better engineer.

7.1 Core concepts to learn (in order)

  1. Client-Server model — HTTP, REST APIs, request/response cycle
  2. Databases — SQL vs NoSQL, indexing, replication, sharding
  3. Caching — Redis, CDNs, cache invalidation strategies
  4. Load balancing — round-robin, consistent hashing, health checks
  5. Message queues — Kafka, RabbitMQ, pub/sub patterns
  6. Microservices vs monolith — trade-offs, when to split
  7. CAP theorem — consistency, availability, partition tolerance

7.2 Practice pattern

# For each system design problem, follow this framework:

1. REQUIREMENTS     → What does the system do? (functional + non-functional)
2. ESTIMATION       → How much data? How many users? QPS?
3. HIGH-LEVEL DESIGN → Draw boxes and arrows (API, services, DB, cache)
4. DETAILED DESIGN  → Zoom into the hardest part (data model, algorithms)
5. BOTTLENECKS      → What breaks at scale? How to mitigate?

# Example: design a URL shortener
- Functional: shorten URL, redirect, analytics
- Scale: 100M URLs, 1000 writes/s, 10K reads/s
- Components: API gateway → shortening service → DB (key-value)
- Key decision: base62 encoding vs hash, read-heavy = cache layer

8. End-to-End Project: From Concept to CI/CD

This is where everything comes together. Here's the full lifecycle of a real project.

Phase 1: Concept & Planning

# Define the problem
- What problem are you solving?
- Who is the user?
- What exists already? (literature review / competitor analysis)

# Write a one-pager
- Problem statement (2–3 sentences)
- Proposed solution
- Success metrics
- Tech stack choices with justification

# Create a GitHub repo with structure
my-project/
├── README.md            # What, why, how to run
├── pyproject.toml       # Dependencies and metadata
├── src/
│   └── my_project/
│       ├── __init__.py
│       ├── model.py
│       └── api.py
├── tests/
│   └── test_model.py
├── docs/
├── .github/
│   └── workflows/
│       └── ci.yml
└── .gitignore

Phase 2: System Design

# Draw your architecture (use draw.io, Excalidraw, or Mermaid)

# Example: ML inference service
┌─────────┐     ┌───────────┐     ┌──────────┐
│  Client  │────▶│  FastAPI   │────▶│  Model   │
│ (React)  │◀────│  Server    │◀────│ (PyTorch)│
└─────────┘     └─────┬─────┘     └──────────┘
                      │
                ┌─────▼─────┐
                │ PostgreSQL │
                └───────────┘

# Define API contracts early
# POST /predict
{
  "input": "text to classify",
  "model_version": "v2"
}
# Response: { "label": "positive", "confidence": 0.94 }

Phase 3: Implementation

# Use branches for each feature
git checkout -b feat/model-inference

# Write tests FIRST (or alongside)
# tests/test_model.py
def test_predict_returns_label():
    result = predict("hello world")
    assert result["label"] in ["positive", "negative", "neutral"]
    assert 0 <= result["confidence"] <= 1

# Run tests locally before pushing
pytest tests/ -v

Phase 4: CI/CD Pipeline

# .github/workflows/ci.yml
name: CI
on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
      - run: pip install -e ".[dev]"
      - run: ruff check src/
      - run: pytest tests/ -v --tb=short

  deploy:
    needs: test
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to server
        run: |
          echo "Deploy step here"
          # e.g., docker build, push to registry, deploy to cloud

Phase 5: Documentation & Release

# Write a proper README
## Project Name
One-line description.

### Quick Start
```bash
pip install my-project
my-project --input "hello"
```

### Results
| Model   | Accuracy | F1    | Latency |
|---------|----------|-------|---------|
| v1      | 87.2%    | 85.1% | 12ms    |
| v2      | 91.4%    | 89.8% | 15ms    |

### Citation
```bibtex
@article{yourname2026,
  title={Your Paper Title},
  author={Your Name},
  journal={arXiv preprint arXiv:2026.xxxxx},
  year={2026}
}
```

The complete flow

Idea → One-pager → System Design → Repo Setup
  → Branch + Code + Tests → PR + Review → CI passes
  → Merge to main → Auto-deploy → Write paper
  → Project website → Push to arXiv → Share
The gap between a student project and a professional one isn't talent — it's process. Follow these steps consistently and you'll build things that last.

TL;DR — Your Learning Path

  1. Set up VS Code + extensions
  2. Learn virtual environments (pick one, master it)
  3. Get comfortable with Git + GitHub + PRs
  4. Write your first LaTeX doc on Overleaf
  5. Deploy a project page on GitHub Pages
  6. Install AI tools (Copilot, Claude Code)
  7. Study one system design problem per week
  8. Build one end-to-end project each semester

Start from step 1 and work your way down. Each skill compounds on the previous one. Good luck.
